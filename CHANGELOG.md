## [0.7.0] - 2026-04-26

### Added

- Phase 1 of the 0.7.0 monitoring & alerting stack, gated on `var.monitoring` per environment. New `terraform/infra/modules/monitoring/` deploys: an `alert_sink` Lambda fronted by a Lambda Function URL that fans out webhook payloads (`{severity, source, summary}`) to Pushover and a self-hosted ntfy server; a `cabal-ntfy` ECS service backed by an EFS access point at `/ntfy`; a `cabal-uptime-kuma` ECS service backed by an EFS access point at `/uptime-kuma`; and a shared public ALB whose default listener action forwards to Kuma behind a Cognito `authenticate-oidc` action and whose host-header rule on `ntfy.<control-domain>` forwards to ntfy without ALB-level auth (ntfy enforces its own bearer-token auth). Severity routing in the Lambda: critical → Pushover priority 1 + ntfy priority 5, warning → ntfy priority 3, info dropped. Partial-failure (one transport up, one down) returns HTTP 207 so callers don't retry-storm.
- `docker/ntfy/Dockerfile` — thin wrapper over `binwiederhier/ntfy`, added to the docker workflow matrix and to the ECR `extra_repositories` list.
- `docker/uptime-kuma/Dockerfile` — thin wrapper over `louislam/uptime-kuma`. The ECS task definition overrides `entryPoint` and runs as UID 1000 to skip the upstream image's `chown -R node:node /app/data`, which fails on EFS access points (access points reject `chown` regardless of caller).
- SSM `SecureString` parameters managed by Terraform with `ignore_changes = [value]` so out-of-band rotation sticks: `/cabal/alert_sink_secret` (auto-generated random for the webhook), `/cabal/pushover_user_key`, `/cabal/pushover_app_token`, `/cabal/ntfy_publisher_token` (operator-populated after creating the Pushover account and bootstrapping the ntfy admin user via ECS Exec).
- `lambda/api/alert_sink/` (formerly `alert_sms/`) — the universal sink Lambda. Reads secrets from SSM with cold-start caching, validates `X-Alert-Secret` with `hmac.compare_digest`, emits per-transport HTTPS POSTs (Pushover form-encoded, ntfy bearer-auth + `Title`/`Priority` headers) using the standard library only; no boto3 SNS/SES dependencies.
- ALB egress rule on the monitoring ALB security group permitting outbound HTTPS to `0.0.0.0/0`. Required for the ALB `authenticate-cognito` action to reach Cognito's hosted UI for the code-for-tokens exchange; without it, `/oauth2/idpresponse` returns 500.
- Two `aws_lambda_permission` resources granting public access to the Function URL: one for `lambda:InvokeFunctionUrl` with `lambda:FunctionUrlAuthType=NONE`, one for `lambda:InvokeFunction` scoped to URL callers via `invoked_via_function_url = true`. Both are required by AWS's URL-gateway auth model; missing either returns 403. The latter argument requires aws Terraform provider ≥ **6.28.0**, which is now the floor for the monitoring module.
- VPC private-zone mirrors of public DNS records so monitoring probes from the private subnet can resolve hosts on the control domain that the private zone would otherwise shadow: `admin.<control-domain>` (CNAME → CloudFront), `uptime.<control-domain>` and `ntfy.<control-domain>` (A aliases → monitoring ALB). Mail-tier hosts (`imap.`, `smtp-in.`, `smtp-out.`) are intentionally not mirrored; the Phase 1 monitor set targets the NLB's public DNS name directly for those probes.
- `docs/monitoring.md` — operator runbook covering Pushover signup, SSM seeding, ntfy admin/token bootstrap via ECS Exec, mobile-app subscription, Kuma admin creation, webhook wiring (with Liquid templating — `{% if heartbeatJSON.status == 0 %}critical{% else %}info{% endif %}`, not Handlebars), the Phase 1 monitor set, secret rotation, and clean disabling.
- `docs/0.7.0/monitoring-plan.md` — multi-phase design doc with the four-phase plan (alert sink + uptime, heartbeats, metrics, logs+tuning), a per-tier golden-signals table, the `var.monitoring` feature-flag pattern, and an "Implementation notes" section capturing the gotchas surfaced by the prod deploy.
- `docs/0.9.0/state-encryption-plan.md` — plan to migrate both Terraform stacks to client-side KMS-encrypted state (TF 1.10+ `encryption` block) so secrets like the Pushover/ntfy tokens can be folded into normal Terraform inputs instead of seeded out-of-band.
- Phase 2 of the 0.7.0 monitoring & alerting stack — heartbeat monitoring for the scheduled jobs that Phase 1's black-box probes can't see (cron-style Lambdas, EventBridge schedules, the in-container reconfigure loop). Gated by the same `var.monitoring` flag — no separate switch. Adds a self-hosted Healthchecks ECS service plus five per-job heartbeats wired through SSM Parameter Store.
- `docker/healthchecks/Dockerfile` — thin wrapper over `healthchecks/healthchecks:v3.10`. Added to the docker workflow build matrix and to the ECR `extra_repositories` list.
- `terraform/infra/modules/monitoring/healthchecks.tf` — Django + uWSGI ECS service on the existing cluster. SQLite on EFS (access point at `/healthchecks`, uid/gid 1000:1000) so check definitions and ping history survive task replacement. `desired_count = 1` with cycle-in-place deployment (single-writer SQLite). Sits behind a new host-header listener rule on the existing monitoring ALB at `heartbeat.<control-domain>` with `authenticate-cognito` (separate `cabal_heartbeat_client` Cognito app client with its own callback URL); the rule reuses the Phase 1 `alb_https_out` egress for the Cognito token exchange, so no SG additions were needed for the Cognito leg. ECS task definition pulls `SECRET_KEY` from SSM via the native `secrets` block; the Healthchecks signup form is gated by a new `var.healthchecks_registration_open` (mirrors `var.monitoring` / `var.backup`: bool, default `false`, sourced from `TF_VAR_HEALTHCHECKS_REGISTRATION_OPEN` in the GitHub environment) so the operator can flip it on for the bootstrap signup and back off without editing the Terraform. Public + private-zone Route 53 aliases for `heartbeat.<control-domain>` mirror the Phase 1 uptime/ntfy pattern (private zone shadows the public zone for the control domain, so without the mirror VPC-internal callers can't resolve it). EFS is mounted at `/var/local/healthchecks-data` (not the upstream image's `/data`) and the container is forced to `user = "1000:1000"` to dodge a separate chown gotcha: the upstream Dockerfile runs `useradd --system hc` (system uid ~999) and `mkdir /data && chown hc /data`, so a bind-mount onto `/data` makes dockerd's copy-up try to chown the host volume path to hc's uid — which the access point's `posix_user` enforcement rejects with `CannotCreateContainerError`. Mounting at a path that doesn't exist in the image skips copy-up entirely; the explicit `user` aligns the runtime uid with the access point's translation so writes succeed. SMTP is wired to `imap.cabal.internal:25` (the existing Cloud Map name for the IMAP tier's local-delivery sendmail) with `EMAIL_USE_TLS=False` — Healthchecks delivers magic-link signup and password-reset mail inbound to itself rather than relaying outbound. No Cognito service user, no DKIM, no auth required, because every recipient is already a Cabalmail-hosted address whose domain is in IMAP's `local-host-names` and whose username is in `/etc/mail/access` (see `docker/shared/generate-config.sh:gen_imap_access`). The envelope FROM is `noreply@mail-admin.${var.mail_domains[0]}` — sendmail's `check_mail` rule does an MX-then-A lookup on the sender's domain and 553-rejects when neither resolves, so the obvious `noreply@<control-domain>` and `noreply@${var.mail_domains[0]}` candidates both fail (control_domain has no apex MX/A; mail-domain apexes are blank by Cabalmail design). The DMARC ingestion infrastructure already provisions `mail-admin.<first-mail-domain>` with full MX/SPF/DKIM/DMARC records (see `terraform/infra/modules/app/dmarc_user.tf`), so re-using that subdomain for system-originated FROM avoids a new DNS namespace. `var.mail_domains` is threaded through `main.tf` into the monitoring module to compose the address. The trade-off is that this Healthchecks instance can only email Cabalmail addresses; magic links to Gmail / external recipients won't deliver. Acceptable for a single-operator setup. `ALLOWED_HOSTS=*` because the ALB target-group health check can't set a custom Host header (it sends `Host: <target-ip>:<port>`), which fails Django's host validation and 400s the probe; hostname enforcement is already done at the ALB layer (the listener rule for `heartbeat.<control-domain>` is the only public path to the target group, and the task SG only accepts traffic from the ALB SG), so Django repeating the check adds nothing.
- `terraform/infra/modules/monitoring/ssm.tf` adds a `random_password`-backed `/cabal/healthchecks_secret_key` (Django secret, rotatable via `terraform taint`) and five `SecureString` placeholders for the per-job ping URLs: `/cabal/healthcheck_ping_certbot_renewal`, `_aws_backup`, `_dmarc_ingest`, `_ecs_reconfigure`, `_cognito_user_sync`. All five follow the Phase 1 placeholder + `ignore_changes = [value]` pattern so the operator pastes the URLs in once after creating each check in the Healthchecks UI and Terraform never overwrites them. (The Phase 2 plan originally included a sixth `_terraform_weekly` placeholder; dropped because the Terraform workflow doesn't run on a cron schedule, so a heartbeat firing only on manual dispatch isn't a useful signal.)
- `terraform/infra/modules/monitoring/backup_heartbeat.tf` + `lambda/api/backup_heartbeat/` — tiny Python Lambda (stdlib only, ~60 lines) wired to an `aws_cloudwatch_event_rule` matching `aws.backup` / `Backup Job State Change` / `state == COMPLETED`. The Lambda reads its ping URL from SSM with cold-start caching, skips silently if the URL is unset or still a `placeholder-…` value, and pings Healthchecks via `urllib.request`. The rule fires regardless of `var.backup` — when the backup plan isn't deployed there are no events to fire, so the Lambda is harmless inert infrastructure.
- Heartbeat instrumentation across five scheduled jobs, all using the same `HEALTHCHECK_PING_PARAM` env-var → SSM-fetch → `urllib` ping pattern with a `value.startswith('http')` guard so placeholder values short-circuit cleanly:
  - `lambda/certbot-renewal/handler.py` — pings at end of `handler` after the cert write and ECS redeployment loop succeed; new `aws_iam_role_policy "healthcheck_ping"` in `modules/certbot_renewal/iam.tf` is gated on `var.healthcheck_ping_param != ""` so monitoring-off environments get neither the env var nor the IAM grant.
  - `lambda/api/process_dmarc/function.py` — pings at the end of both successful return paths (no-messages early return and post-batch-write); IAM policy in `modules/app/dmarc.tf` extends the existing `ssm:GetParameter` resource list via `compact()` so the ping ARN is appended only when the param name is non-empty.
  - `lambda/counter/assign_osid/function.py` — pings after `refresh_containers()`. `modules/user_pool/counter.tf` IAM policy migrated from heredoc JSON to `jsonencode` to support the conditional `ssm:GetParameter` statement append.
  - `docker/shared/reconfigure.sh` — bash `ping_healthcheck` helper called at the end of `regenerate()`. Reads `HEALTHCHECK_PING_URL` (injected as an ECS task-definition `secret` from `/cabal/healthcheck_ping_ecs_reconfigure` only when monitoring is enabled, via a `local.healthcheck_secrets` `concat`-into-secrets pattern in `modules/ecs/task-definitions.tf`), case-matches on `http://*` / `https://*`, fires `curl -fsS -m 5`, and tolerates failure. The mail-tier execution role's existing `/cabal/*` SSM read covers the new param without extra IAM.
  - `backup_heartbeat` Lambda (above) handles the AWS Backup case via EventBridge rather than instrumenting AWS Backup's plan resource, since AWS Backup doesn't accept synchronous webhooks.
- Heartbeat consumers thread through a `healthcheck_ping_param` (string, default `""`) variable on each module instead of leaking `var.monitoring` directly. Root `terraform/infra/main.tf` builds the four canonical names as locals (`hc_ping_certbot`, `hc_ping_dmarc`, `hc_ping_assign_osid`, `hc_ping_ecs_reconfigure`) gated on `var.monitoring ? "/cabal/healthcheck_ping_…" : ""`, so flipping `monitoring` off cleanly drops the env var, the IAM grant, and (for the mail tiers) the ECS `secret` entry. The string names match what the monitoring module creates by convention; no cross-module reference is needed, which keeps consumer modules decoupled from `count = 0` monitoring.
- Healthchecks bootstrap and per-job check creation are documented as a new "Phase 2" section (steps 11–14) in `docs/monitoring.md`: the magic-link first-boot flow keyed off `TF_VAR_HEALTHCHECKS_REGISTRATION_OPEN` (added to the §2 GitHub-environment-variables table alongside `TF_VAR_MONITORING`), a `createsuperuser` ECS-Exec fallback if the operator wants to bootstrap before adding a Cabalmail address, the recommended grace windows per check, the Healthchecks → `alert_sink` webhook integration template (`{"summary": "Missed heartbeat: $NAME", "severity": "critical", "source": "healthchecks/$NAME"}`), the acceptance checklist, individual-heartbeat pause via either the UI or a non-`http` SSM value, and a Phase 2 troubleshooting block covering the dockerd-copy-up chown failure (with the `/var/local/healthchecks-data` mount-path workaround), the `ALLOWED_HOSTS=*` rationale for ALB health checks, the ping-URL caching behavior on Lambda cold start vs ECS task start (and the `--force-new-deployment` workaround for the latter), and the `var.backup`-must-be-true interaction with `backup_heartbeat`.

- Phase 3 of the 0.7.0 monitoring & alerting stack — Prometheus, Alertmanager, Grafana, plus three cluster-scope exporters (cloudwatch, blackbox, node) and seven initial alert rules. Gated by the same `var.monitoring` flag — no separate switch.
- `docker/prometheus/`, `docker/alertmanager/`, `docker/grafana/` — thin wrappers over upstream `prom/prometheus:v3.5.0`, `prom/alertmanager:v0.28.1`, and `grafana/grafana:11.4.0`. Prometheus and Alertmanager get small entrypoint shims that `sed`-substitute runtime placeholders (control domain, environment label, alert_sink Function URL, alert_sink shared secret) into baked-in config templates before exec; Grafana relies on file-based provisioning (`docker/grafana/provisioning/datasources/`, `dashboards/`) for the Prometheus datasource and four bootstrap dashboards (Mail Tiers, AWS Services, API Gateway & Lambda, Frontend). Configs and rules are baked in deliberately — rebuild + redeploy on change is the upgrade path; no SSM hot-reload, since the alert-tuning cadence (`docs/0.7.0/monitoring-plan.md` §"Tuning discipline") aligns with the per-PR review pattern.
- `docker/cloudwatch-exporter/`, `docker/blackbox-exporter/`, `docker/node-exporter/` — wrappers over the upstream `prom/cloudwatch-exporter:v0.16.0`, `prom/blackbox-exporter:v0.27.0`, and `prom/node-exporter:v1.9.1` images. cloudwatch_exporter scrapes AWS/Lambda, AWS/DynamoDB, AWS/EFS, AWS/ECS, AWS/ApiGateway, AWS/ApplicationELB, AWS/CertificateManager, and AWS/Cognito; the metric set is intentionally small to keep the GetMetricStatistics call-rate down. blackbox_exporter ships modules `http_2xx`, `tcp_connect`, `tcp_tls`. node_exporter ships unmodified — the deviation is in the ECS service that runs it (DAEMON-strategy with host bind-mounts), not the image.
- `terraform/infra/modules/monitoring/discovery.tf` — new Cloud Map private DNS namespace `cabal-monitoring.cabal.internal` with one `aws_service_discovery_service` per metrics component. Prometheus uses DNS-SD `type: A` queries against these names so scrape targets follow task replacements automatically. Separate from the mail-tier `mail.cabal.internal` namespace so monitoring DNS records can never collide with mail-tier records.
- `terraform/infra/modules/monitoring/prometheus.tf` — single ECS task; TSDB on a new EFS access point at `/prometheus` (uid/gid 65534 to match the upstream image's `nobody` user). 30-day retention, `--web.enable-lifecycle` so configuration reloads can be triggered out-of-band via `POST /-/reload` from inside the task. `desired_count = 1` with cycle-in-place deployment (single-writer TSDB). No public surface — operators reach Prometheus via Grafana's data-source proxy or `aws ecs execute-command` port-forwarding.
- `terraform/infra/modules/monitoring/alertmanager.tf` — single ECS task; silences and the notification log on a new EFS access point at `/alertmanager`. Posts to the existing `alert_sink` Lambda Function URL with a Bearer token (the shared secret is injected via the ECS `secrets` mechanism from `/cabal/alert_sink_secret`, scoped to the Alertmanager execution role's `ssm:GetParameters`). The Lambda accepts both `X-Alert-Secret` (Phase 1/2 callers) and `Authorization: Bearer` (Phase 3); see the `lambda/api/alert_sink/function.py` change below. Alertmanager's webhook config doesn't allow custom JSON bodies, so the bridging logic — pulling severity/summary/source off the first alert in `alerts[]`, downgrading resolved-alert severity to `warning` so a recovery doesn't re-page on Pushover — lives in the Lambda. Routing: critical → 24h `repeat_interval` (re-page on stuck criticals once a day), warning → 6h. Inhibit rule on `CertExpiringSoonCritical` suppresses the warning variant for the same domain.
- `terraform/infra/modules/monitoring/grafana.tf` — single ECS task; SQLite store on a new EFS access point at `/grafana` (uid/gid 472 for the upstream image's `grafana` user). Reachable at `https://metrics.<control-domain>/` behind the existing monitoring ALB with a new host-header rule (priority 120, after Phase 1's ntfy=100 / heartbeat=110) and its own Cognito client. Grafana runs in anonymous-Viewer mode at the ALB seam — Cognito-authenticated users land as Viewers; admin actions still require the local admin password from `/cabal/grafana_admin_password` (auto-generated `random_password`, `ignore_changes` so rotation sticks). Provisioning baked into the image: a Prometheus datasource pointing at `http://prometheus.cabal-monitoring.cabal.internal:9090` and four dashboards under the **Cabalmail** folder.
- `terraform/infra/modules/monitoring/exporters.tf` — three ECS services for the cluster-scope exporters. cloudwatch_exporter and blackbox_exporter are single-task with `desired_count = 1` and a shared `cabal-monitoring-exporters` security group (Prometheus-only ingress, broad egress). cloudwatch_exporter has an IAM task role policy granting `cloudwatch:ListMetrics`, `cloudwatch:GetMetricData`, `cloudwatch:GetMetricStatistics`, `tag:GetResources` on `Resource: "*"` (the exporter discovers metrics across all configured namespaces on each scrape, so wildcard is the only viable scope). node_exporter is **DAEMON-strategy** (`scheduling_strategy = "DAEMON"`, no `desired_count`) so each cluster instance runs exactly one task; uses `network_mode = "host"` and bind-mounts host `/proc`, `/sys`, and `/` read-only with `--path.rootfs=/host` so it reports the EC2 host's view rather than the container's.
- node_exporter framing deliberately deviates from the §3.2 plan, which called for it as a per-tier sidecar in each mail-tier task definition. A daemon service yields one set of host metrics per EC2 instead of three duplicates per host, avoids changing every mail-tier task definition (a destructive operation per `docs/0.7.0/monitoring-plan.md` §6 lesson "stable-flag discipline for sidecars"), and slots cleanly into the existing `cabal-ecs-instance-sg` which already permits VPC-internal ingress so Prometheus can reach `host:9100` without an SG additions. Documented in the metrics.tf comment block and in `docs/monitoring.md` § "Phase 3 — deferred items".
- `lambda/api/alert_sink/function.py` — accepts the secret via either `X-Alert-Secret` (Kuma, Healthchecks) or `Authorization: Bearer` (Alertmanager's `http_config.authorization`). Translates Alertmanager's native webhook v4 body — `{status, alerts: [...], groupLabels, ...}` — into the existing `{severity, summary, source}` shape: pulls severity/alertname/instance from the first alert's labels and synthesizes a single push with a `(+N more)` suffix when the group bundles multiple alerts. Resolved status flips severity to `warning` so a recovery pings ntfy without re-paging on Pushover.
- `terraform/infra/modules/monitoring/cognito.tf` adds `aws_cognito_user_pool_client.grafana` (`cabal_metrics_client`) with its own `https://metrics.<control-domain>/oauth2/idpresponse` callback and `/` logout, mirroring the existing kuma/healthchecks per-host pattern.
- `terraform/infra/modules/monitoring/alb.tf` adds the `aws_lb_listener_rule.grafana` host-header rule on `metrics.<control-domain>` at priority 120, with its own `authenticate-cognito` action pointing at the metrics client and a forward to the new `cabal-grafana` target group.

Existing ALB egress rule from Phase 1 already covers Cognito; existing `cabal-ecs-instance-sg` from the ecs module already permits VPC-internal ingress for the node_exporter daemon.

### Removed

- `TF_VAR_ON_CALL_PHONE_NUMBERS` and the SMS-via-SNS alerting path. AWS toll-free SMS provisioning was slow and opaque, and SES email can't alert on our own mail outage. Replaced with the Pushover + ntfy push-notification fan-out described above.

### Operational notes

- The monitoring ALB requires ≥2 AZs. Production has two AZs; dev and stage have one each, and the `cidrsubnet` math in the VPC module makes adding a second AZ to those environments destructive (every subnet is renumbered). Phase 1 was therefore deployed directly to prod. Future work: have the monitoring module own a small extra public subnet in a second AZ instead of leaning on the VPC's per-AZ list.
- Phase 1 SSM seeding is manual, deliberately. The placeholder + `ignore_changes` pattern keeps real secrets out of state and out of `terraform.tfvars`. The 0.9.0 plan replaces this once state encryption is in place.
- Phase 3 image build must precede the first Terraform apply that flips `var.monitoring` on after this release lands. The new ECR repos are populated by the next Docker workflow run; if Terraform applies first, the ECS services for Prometheus / Alertmanager / Grafana / cloudwatch-exporter / blackbox-exporter / node-exporter sit pending until the `sha-<first-8>` tag exists. Existing services (Kuma, ntfy, Healthchecks) keep running throughout. Document order: Docker workflow → Terraform workflow.
- Phase 3 deliberately does **not** ship the per-tier mail-tier sidecar exporters (`dovecot_exporter`, `postfix_exporter`, `opendkim_exporter`) or the `MailQueueGrowing` / `MailDeliveryFailureRate` alert rules described in §3.2 / §3.3 of the plan. The blocker is the postfix_exporter / sendmail log-format mismatch — Cabalmail uses Sendmail with `/var/log/maillog` lines like `sm-mta[…]: ABCD: from=…, stat=…`, but postfix_exporter parses Postfix's `qmgr[…]: ABCD: from=…, size=…, nrcpt=…, (queue active)` shape; making it useful would need either a per-line adapter, a fork, or a switch to Loki+LogQL log-derived metrics. The natural place to settle that trade-off is alongside Phase 4 log aggregation rather than under the same PR as the cluster-scope metrics. node_exporter likewise deviates from §3.2's per-tier sidecar framing in favor of a single DAEMON-strategy ECS service (one task per EC2 host instead of three duplicates per host) — fewer tasks, cleaner dashboards, and zero changes to mail-tier task definitions which would otherwise force task replacement.

## [0.6.4] - 2026-04-22

### Added

- Apple client: the folder message list now refreshes on a 60-second wall-clock timer while it is on screen, in addition to the existing IMAP IDLE push. IDLE usually delivers new mail within seconds, but long-lived IDLE sockets can stall silently — iOS suspends idle connections while the app is foregrounded but network-idle, cellular ↔ WiFi handoffs drop the stream without surfacing an error, and some middleboxes time out TCP idle after a few minutes. The net effect was that users had to pull-to-refresh to see new messages even with the app in the foreground. A second SwiftUI `.task` on `MessageListView` sleeps 60 s and calls `MessageListViewModel.refresh()`; `.task` auto-cancels on `.onDisappear` so the timer starts and stops together with the IDLE watcher and doesn't hold a connection open for a mailbox the user isn't looking at.
- Apple client: the app icon now shows a badge with the Inbox unread count on iOS / iPadOS / visionOS (home-screen icon) and macOS (dock tile). `AppState` owns an independent poller that runs while signed in, requests `.badge` authorization via `UNUserNotificationCenter` on first start (silently no-ops on denial or repeat calls), polls `STATUS (UNSEEN)` on `INBOX` every 60 seconds, and pushes the count through `UNUserNotificationCenter.setBadgeCount(_:)`. The poll is independent of which folder the user is viewing, so the badge stays current even while Drafts/Sent are on screen. Started at the `.signedIn` transition of both `signIn()` and `restoreIfPossible()`; stopped (and the badge cleared) at the start of `signOut()` so the icon doesn't keep showing the previous user's count. - **Let's Encrypt production CA in every environment.** `lambda/certbot-renewal/handler.py` no longer branches on `USE_STAGING`, and the `certbot_renewal` Terraform module drops its `prod` input variable. Staging certs (previously used for non-prod environments) aren't trusted by iOS/macOS root stores, so an Apple client hitting `smtp-out.<control_domain>` on port 465 (TCP passthrough, Dovecot presents the Let's Encrypt cert directly) couldn't complete the implicit-TLS handshake. The certbot Lambda re-issues against production on its next run; force a manual invocation after deploy if you want the swap immediately rather than on the scheduled renewal.
- **Certbot Lambda `image_uri` rotates per deploy.** The module had `image_uri = "${repo_url}:latest"` hardcoded, which Terraform treats as a string that never changes — the Lambda kept running whichever image had been pushed the first time it was created, even when CI pushed a new `:latest` tag. The module now takes an `image_tag` input wired from `/cabal/deployed_image_tag` (the same SSM parameter the ECS task definitions consume), so the Lambda's image reference rotates on every deploy.
